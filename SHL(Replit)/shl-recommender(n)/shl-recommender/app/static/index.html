How It Works

Architecture and design decisions behind the SHL Assessment Recommender

1

You Describe the Role

Tell the assistant about the role you're hiring for — job title, skills needed, seniority level. Vague descriptions trigger a clarifying question.

2

BM25 Retrieval

Your query (with synonym expansion for recruiter vocabulary) is matched against 377 SHL assessments using BM25 keyword search with metadata boosting.

3

LLM Grounding

The top 20 candidates are passed to an LLM (Groq/Gemini) with a strict grounding rule: it can only recommend URLs present in the retrieved candidates.

4

Verification

Every URL returned by the LLM is verified against the master catalog. Hallucinated URLs are dropped silently — only real SHL assessments appear.

5

Stateless Continuity

The API stores no state. Shortlist continuity across turns is achieved by embedding a hidden URL marker in the assistant reply, which travels with the history.

6

Safety Guardrails

A deterministic pre-filter blocks prompt injection, off-topic requests (legal, medical, coding), and general hiring advice before the LLM is ever called.
